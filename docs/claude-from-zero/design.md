## Group: Why Claude Code

**1. 为什么是 CC 而不是"龙虾"**

> 过渡：这里要回答一个自然的疑问——网页版 Agent（如 Kimi Computer、Manus 等）已经能制作完整应用了，为什么还要学习 Claude Code？

从三个维度回答。

**维度一：高质量的输出需要高质量的输入——"垫"**

既然 AI 是生成式的，需要输入和输出，那么高质量的输出必然依赖高质量的输入。这里存在一个关键动作叫"垫"：由人把控重要的中间环节，在关键节点维护高质量的输入，才能得到良好的最终输出。

**例子**：先用 gpt-image-v2 生成一张高质量照片，再用可灵/即梦生成视频。每一步的"垫"决定了最终品质。

**维度二：专业程度需要掌控细节**

想把一件事办到专业水平，需要学习及控细节。纯脑力和体力外包，不会给你带来任何经验积累。

**维度三：集成式开发环境减少上下文切换损耗**

IDE（集成式开发环境）把所有工具打包在一个应用中，不需要切换工具来完成任务。频繁的上下文切换是效率杀手——就像工作时有人不断打断你添加新任务。Claude Code 的集成式工作环境避免了这种损耗。

## Group: LLM 是基于概率的

**扉页设计**
- 标题：LLM 是基于概率的
- 核心观点：它不是查表，不是 if-else，是万亿参数在掷骰子
- 视觉：一个巨大的骰子，每个面写着不同的 token

---

**1. 小朋友怎么学习**

> 过渡：先从人脑的学习说起，因为神经网络的设计灵感正来源于此。

"学习"很抽象。教小孩认"什么是苹果"，你不知道他如何记住的，但大概是通过五感：气味、颜色、形状等组合识别。

这里需要一幅对比图：传统方法 vs 机器学习方法。

- **传统方法**：无数 if-else——if 红色 and 圆形 and 有香味 then 苹果。规则写不完，且无法处理"边缘情况"。
- **机器学习方法**：不预设规则，让系统自己从数据中找到规律。

但小孩记住苹果时，脑海里真的是一个一个记住颜色、形状、味道的吗？不是的。这些信息被并行处理、综合感知。

---

**2. 神经网络的结构**

> 过渡：小孩的学习是"并行综合感知"，神经网络的设计正是模拟这一点。

展示一张神经网络结构图：

- 左侧输入层：X 属性（颜色）、Y 属性（形状）、Z 属性（气味）……所有关于苹果的属性并行输入
- 中间隐藏层（多层）：方框把隐藏层圈起来，标明是黑盒（未知运算）
- 右侧输出层：单个节点标注了"苹果"

**解释**：所有属性并行处理。有人特别在意颜色，有人特别在意形状（比如凸出来的柄）。但没人弄得清楚，大脑里到底在算什么。这种"不知道具体怎么运算，但能得出正确结果"的机制，我们叫它"机器学习"。

神经网络是机器学习的具体方法之一，结构类似于人脑的神经结构，因此得名。

---

**3. 从神经网络到大语言模型**

> 过渡：理解了神经网络的结构，接下来看规模带来的质变。

大语言模型本质上是一个超大规模的神经网络。输入层、输出层、隐藏层的数量极其庞大，总参数可达万亿级别。

**为什么叫"基于概率"？**

参数量达到恐怖级别后，我们没有办法通过传统统计学方法去了解每一个参数代表什么特性。任何一个输入层的微小扰动，都可能严重影响计算结果。它对我们来说是一个"黑盒"——输入进去，输出出来，中间发生了什么不可解释。

这点可以类比天气预报，就算我们建了非常多洋流监测点、风向标，但 24 小时以后的天气还是预测不准（甚至 24 小时以内也经常出错）。

正因为这种不可解释性，它的输出不是"确定的事实"，而是"概率上的最可能结果"。

---

**4. AI 是怎么学习的**

> 过渡：知道了结构，再看它"怎么学会"一件事的——也就是训练过程。

节点本身有个数值，叫做**输入**。节点之间有一条边，边上是一个任意数字，叫做**权重**。

某个节点的值 = 所有输入 × 对应权重，加权求和得到。经过一层一层计算，最终得到一个结果。

这个结果首先是一个数字。假设输出节点代表"苹果"这个概念：

- 输出 0.999 → 几乎肯定是苹果
- 输出 0.7 → 可能是个苹果
- 输出 0.3 → 应该不是

**训练的过程**：

假设我们手里真的有一个苹果，输入"红色、圆形、有香味"，模型输出 0.7。它"猜对了一半"，但和正确答案（1.0）之间的差距 0.3 就是**损失**。

现在调整模型中的参数（权重）。调参后输出改变，如果向 1.0 靠近，说明调对了，模型"学到了"。

这里需要一张梯度下降示意图。

**梯度下降的通俗解释**：

想象一个起伏的平面，平面越高 = 模型越没学到东西。把一个小球放在平面上，给它一个初始推力，让它往不同方向滚动。有时滚向更高的地方（无效调参），有时滚向低洼处（有效学习）。

小球滚到一个坑里停住，再去推它也不动了——这时模型就"训练完成"了，可以对外发布。这个坑里的值不一定是 1.0，可能是 0.95，但 0.95 已足以断定"这是苹果"。

**2. Token：LLM 的"原子"**

> 过渡：要理解 Agent，先退回来看神经网络最基本的输入输出单位。

大语言模型的输入和输出都是文本。但刚才说输入是"不同属性"，文本怎么对应属性？

这里需要展示**分词（Tokenization）**的过程：

一段话被拆分成不同的词/片段，每个片段叫做一个 **token**。token 经过特定处理（向量化）后，成为神经网络的一系列节点输入。

神经网络的输出也是一个个 token，当模型持续输出 token，把它们累加起来，就得到了句子。

但如果没有输入，还能生成东西吗？显然是不行的。这就是"生成式模型"的含义：基于输入，不断生成新东西。

---

**3. Chat：让 LLM 变聪明的工程技巧**

> 过渡：既然 LLM 只是生成句子，为什么 ChatGPT 3.5 显得这么智能？

两个关键因素：

**微调（Fine-tuning）**：专家用大量"问题-正确回答"对训练模型。如果说错了（手里是苹果，他说梨），就纠正。如果步骤错了（XYZ 做成了 XZY），就告诉正确步骤。

**涌现（Emergence）**：2012 年后算力爆发，模型参数规模增大（神经网络变大），突然捕获了小型神经网络抓不住的特征，能力突飞猛进上涨。

从 ChatGPT 3.5 开始，我们可以用对话形式与 LLM 交互。你输入的句子叫做**提示词（Prompt）**。

**背后的原理**：

模型本质上还是输入→输出。

这时候有个大聪明，在工程上巧妙地把角色分为两种：用户和助手，然后在对话的时候这样输入：

```
用户询问了问题，你来补全"..."空白处内容。

用户：今天天气怎么样？
助手：...
```

模型会接着"助手："后面生成回答，这叫做**对话补全**。

如果模型回复："晴天，但我查了天气预报说一会儿可能下雨，所以我要诚实地告诉用户，他的出行计划可能泡汤了……"——这个内心戏叫做**推理（Reasoning）**。

推理内容的长度可以设置，叫做**推理预算**。如果供应商突然减少这个预算，模型表现下降，叫做**降智**。

**关键提醒**：推理可能导向正确结果，但推理本身不一定是真的。比如模型说"我查了天气预报"，只是碰巧生成了这几个 token，实际上没查，导出的结果显然是错的。模型一本正经胡说八道，这叫**幻觉**。

**上下文**：

如果模型回答“你在上海还是北京”，我们再次询问“上海”时，大语言模型怎么知道我们是在询问天气呢？

实际上，所有的会话历史都发送给了大模型：

```
你补全最后一句的空白处内容就可以。

用户：今天天气怎么样？
助手：你在上海还是北京？
用户：上海
助手：...
```

当再次询问"今天天气怎么样"时，ChatGPT 会把上一轮的问题、回答，连同新一轮的问题一起传给模型。这个被传递的可能影响此轮询问的内容就是**上下文**。

上下文长度有限制——就像人脑同时处理的任务有上限。塞太多会**上下文溢出**，模型直接报错罢工。

钱钟书《围城》二十来万字，差不多是 26 年年初的主流模型的上下文上限。

**4. Agent：让概率输出尽可能确定的工程框架**

> 过渡：理解了 Chat 的原理，现在回答"Agent 是什么"。

这里展示经典的 ReAct 模型图。

经常听到普遍解释：Agent 是基于目标的、能感知环境、选择工具执行、观察结果并循环的框架。但太抽象了，谁也看不懂。

所以回到本次分享的核心，换种方向解释 Agent 的作用——从"概率"出发。

为什么需要 Agent？答案很简单：**通过工程化手段，让 LLM 的概率输出尽可能确定下来**。不是要求每个字都确定，而是需要引导、需要方向。

---

引导方向很简单，该提示词就行：

```
你是村上春树风格写作顾问，改写我的文章，以第一人称视角，带有疏离感的冷静叙述，简洁、克制的短句为主
这是我的文章：xxxx
```

---

直接发给 LLM 他也能生效并完成任务，只是每次都要写这么长串提示词太麻烦了，不如放到 Agent 的角色卡片里和 Agent 绑定在一起。

在这个例子里，你通过打磨提示词，获得不错的效果，叫做**提示词工程（Prompt Engineering）**——23 年特别火的"怎么好好和模型说话"的学问。

好的提示词遵循框架，比如说**CRISPE 框架**：

| 字母  | 英文                  | 中文       | 含义                                     |
| ----- | --------------------- | ---------- | ---------------------------------------- |
| **C** | **C**apacity and Role | 能力与角色 | 让 AI 扮演什么角色（如"资深前端工程师"） |
| **R** | **R**eason            | 原因/理由  | 为什么要做这件事，业务背景是什么         |
| **I** | **I**nsight           | 洞察/背景  | AI 需要了解的相关信息、上下文、数据      |
| **S** | **S**tatement         | 陈述/指令  | 具体要 AI 执行什么任务                   |
| **P** | **P**ersonality       | 个性/风格  | 回答的语气、风格、人格特征               |
| **E** | **E**xperiment        | 实验/尝试  | 要求提供多个方案或从不同角度分析         |

**示例**（用 CRISPE 写提示词）：

> 你是一位有 10 年经验的 **React 性能优化专家**（C）。我们的电商网站在移动端首屏加载超过 4 秒，导致转化率下降（R）。当前技术栈是 Next.js 14 + Tailwind，主要页面包含大量商品图片和筛选组件（I）。请给出 **3 个不同侧重点的优化方案**（E），每个方案需包含具体代码示例和预期提升数据（S），语气要直接、不废话，像资深工程师给 junior 的 code review 一样（P）。

但每次都这么写累死个人，干脆把这个东西做到 LLM 内部算了，这和我们讲的 Agent 角色卡片实际上是一回事。刚才的提示词改写成角色卡片，就这样写：

```
# 角色：村上春树风格写作顾问
你是一位专精村上春树写作风格的文学顾问...

第一人称视角，带有疏离感的冷静叙述，简洁、克制的短句为主

## 输出示例
**原文**：我很难过，失恋了。
**村上化改写**：说实话，我并不觉得特别难过...
```

这叫**系统提示（System Prompt）**——内置于模型或Agent的角色设定。

---

那 Agent 的**工具调用** 又是什么？

简单来说工具是指一个应用、程序或函数。介绍展示AI 操作电脑的时序图。

用户："帮我查昨日出海订单"
→ 模型推理："我需要调用查询订单工具"
→ 发起工具调用
→ 工具层执行函数
→ 返回结果："42 单"
→ 模型推理："我得到了结果，可以直接回答用户了"

---

**ReAct 循环再理解**：

回到 ReAct 循环图，DeepSeek 说的"专门针对 Agent 长程任务做的优化"是什么？

模型在一个任务过程中，会多次调用工具，得到特定结果，然后重新推理。相比"输入→直接输出"，完成任务时模型反复调用工具、尝试得出更好结果，这个循环就是 **ReAct 循环**。

> 如果今天只能记两个观点：一个是"概率"，另一个就是"ReAct 循环"。

**错误恢复**：某一次工具调用出错，一般可以在下一次推理中被发现并修正。

**长程任务**：Agent 执行了特别长时间（可能几小时）的任务。

**5. "高级概念"**

> 过渡：理解了 Agent 的核心循环，现在来看它如何与外部世界打交道——以及一些衍生的工程概念。

Agent 是封闭系统，工具定义在内部。要和外部打交道，一定需要网络请求。

假设有一个"查询天气"的函数，为了让模型能调用它，需要做一层协议封装，这层协议叫 **MCP（Model Context Protocol）**。

---

**Agentic vs Chat**：

- 有工具调用、有 MCP 等高阶能力 → **Agentic**
- 只和终端用户做问答，没有工具调用 → **Chat**

---

展示游戏小镇的例子：每个角色是一个独立 Agent，有独立目标，能各自完成任务。角色之间可以沟通，Agent 之间的通讯协议叫 **A2A（Agent to Agent）**。

---

**Skills（技能）**：

Agent 执行某个任务做得很好，把流程总结成文档，告诉它"以后做 X 就参考这份文档"——这份文档就是**技能**。

**RAG（检索增强生成）**：

简单流程可以写成文档，但海量知识无法全部枚举。比如全国 6000 万家公司，不可能全部写入提示词。

解决方案：建立数据库，每次提问时先检索相关资料，把检索结果和用户问题一起传给 LLM。这叫 **RAG**。

---

至此，LLM 和 Agent 相关的关键词就结束了。

**6. OpenClaw 和 Claude Code**

> 过渡：理解了 Agent 的本质和各种工程概念，现在来看一个具体的实现——以及 Claude Code 在其中的位置。

**OpenClaw**（看图）：

- 通用 Agent，内部是 ReAct 循环
- 能拆解任务并执行得到结果
- 附加功能：系统提示、引导系统、记忆、文件系统、权限控制、技能、聊天渠道等
- 这些功能围绕"让通用 Agent 更像人格化产品"打造，但本质仍是一个 Agent

![OpenClaw Architect](https://mgear-image.oss-cn-shanghai.aliyuncs.com/image/other/20260428031808284.png)

---

**Claude Code 的定位**：

Claude Code 虽然叫 Coding Agent，但本质上是通用 Agent，也可以做非代码类的事情。这和 OpenClaw 是通用 Agent 但也能写代码是一样的道理。

---

最后一个概念：**Harness = Product - Agent**。

Agent 指最基础的 ReAct 循环。除此之外的所有功能——打造产品的基础能力、和业务价值挂钩的技能、插件等等——通通叫 Harness。

Harness + Agent = 最终产品。

可以想象未来的 AI 产品，比拼的不是谁的框架更高级，而是应该围绕 Agent 做的工程化有多好、给的技能和插件如何提升业务效率，来评价其 Harness 能力有多强。

我们也别忘了Harness 的本质工作，是让 Agent 得到确定性的（可用的）产出。
